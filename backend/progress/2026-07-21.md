## 21:30 — Минимальный JWT-логин (POST /api/auth/login)

- Что сделано: Добавлена минимальная реальная авторизация — только логин для уже существующих сидированных пользователей (без регистрации, без customer/farmer флоу — это остаётся отдельным Этапом 2 по ТЗ, раздел 23). `GetByEmailAsync` в IUserRepository/UserRepository; `ITokenService`/`TokenService` (MarketTJ.Infrastructure/Security) — генерирует JWT (HMAC-SHA256) с claims NameIdentifier/Email/Role; `IAuthService`/`AuthService` — email → BCrypt.Verify(PasswordHash) → проверка IsActive → токен; при любой из трёх причин отказа (нет email/неверный пароль/неактивен) — одно и то же сообщение "Неверный email или пароль" (не палим деталей снаружи), ErrorType.Unauthorized → HTTP 401 через уже существующий ApiControllerBase.HandleResult. `LoginRequestDto`/`LoginResponseDto` (без PasswordHash в ответе). `AuthController` (`POST /api/auth/login`). В Program.cs — `AddAuthentication().AddJwtBearer(...)` + `app.UseAuthentication()` перед уже существующим `UseAuthorization()` (Authentication отсутствовал в пайплайне вообще, только UseAuthorization без UseAuthentication — то есть даже если бы где-то стоял `[Authorize]`, он бы не работал); JWT Bearer security definition добавлена в Swagger (как и планировалось в старом комментарии в коде).
- Файлы: MarketTJ.WebApi/{MarketTJ.WebApi.csproj, appsettings.json, Program.cs, Controllers/AuthController.cs}, MarketTJ.Application/{MarketTJ.Application.csproj, DependencyInjection.cs, Interfaces/Services/{IAuthService.cs,ITokenService.cs}, Services/AuthService.cs, Dto/AuthDto/{LoginRequestDto.cs,LoginResponseDto.cs}, Interfaces/Repositories/IUserRepository.cs}, MarketTJ.Infrastructure/{MarketTJ.Infrastructure.csproj, DependencyInjection.cs, Security/TokenService.cs, Persistence/Repositories/UserRepository.cs}
- Решения/примечания: Пакеты — Microsoft.AspNetCore.Authentication.JwtBearer 10.0.10 (WebApi), System.IdentityModel.Tokens.Jwt 8.19.2 (Infrastructure — нужен отдельно от WebApi, т.к. TokenService физически лежит в Infrastructure и не наследует пакеты WebApi), BCrypt.Net-Next пришлось добавить и в Application.csproj отдельно (был только в Infrastructure, а AuthService/пароль-верификация — в Application). Эта версия Swashbuckle/Microsoft.OpenApi (v2, идёт с .NET 10) поменяла namespace `Microsoft.OpenApi.Models`→`Microsoft.OpenApi` и убрала прежний API `OpenApiSecurityScheme.Reference`/`OpenApiSecurityRequirement` в старом виде — упростил до одной AddSecurityDefinition без AddSecurityRequirement (не критично, влияет только на удобство Swagger UI, не на функциональность). JWT-секрет (случайный 48-байтовый base64) лежит в appsettings.json открытым текстом — по уже принятой в проекте практике (там же исторически лежит пароль от Postgres), не идеально для security best practice, но соответствует существующей конвенции репозитория; ExpiryMinutes=120, без refresh-токена (раздел 16 ТЗ помечает refresh как "рекомендуется, необязательно для MVP"). CORS не добавлялся — вместо него Vite dev-proxy на фронтенде (см. Frontend/progress/2026-07-21.md), так что preflight не возникает в dev-режиме; для прода потребуется либо CORS, либо единый origin — вне текущего скоупа. Полный сквозной прогон через Playwright с реально поднятым `dotnet run` (Postgres/Redis локально) подтвердил: верный логин → JWT с корректными claims; неверный пароль → 401 с ожидаемым сообщением; `dotnet build` всего решения — 0 предупреждений/ошибок.
