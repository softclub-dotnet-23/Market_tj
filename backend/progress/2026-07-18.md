## Unit-тесты для всех 30 Application/Services/*Service.cs

- Что сделано: по `generate-service-tests-prompt.md` — до начала задачи
  `Application.Tests/Services/` был полностью пуст (только `.gitkeep`, тестов
  не было вообще ни для одного сервиса, «эталонного» `DiscountRuleServiceTests`
  не существует). Тест-фреймворк в проекте — xUnit (`xunit` 2.9.3), mocking-
  framework отсутствовал — добавлен `Moq` 4.20.72 (пакет уже был в локальном
  NuGet-кэше) в `MarketTJ.Application.Tests.csproj`. Для каждого из 30
  `{Entity}Service.cs` написан `Application.Tests/Services/{Entity}ServiceTests.cs`:
  зависимости (репозитории, `ILogger<T>`) замоканы через `Mock<T>`, сама логика
  сервиса не мокается. Покрыты: `GetAllAsync` (список/пусто/исключение),
  `GetByIdAsync` (найдено/NotFound/исключение), `CreateAsync` (валидные данные,
  отдельный тест на каждое правило валидатора, отдельные тесты на
  FK/уникальность/бизнес-правила, исключение репозитория), `UpdateAsync`
  (аналогично + запись не найдена), `DeleteAsync` (успех/NotFound/исключение).
  Кэш-сценарий из промпта не применялся ни к одному сервису — ни один из 30
  `{Entity}Service` не принимает `ICacheService`/`IMemoryCache` в конструкторе
  (кэширование, упомянутое в разделе 8.17 ТЗ для AppSetting, в реальный код ещё
  не добавлено).
- Файлы: 30 файлов `*ServiceTests.cs` (по одному на сервис) +
  `MarketTJ.Application.Tests.csproj` (добавлен `PackageReference Moq`).
- Порядок реализации (один коммит на сервис — `git commit -m "add tests for
  {Entity}Service"`): User, FarmerProfile, CustomerProfile, CourierProfile,

  FarmerDocument, AuditLog, ReportedListing, RefundRequest, DeliveryZone,

  все тесты зелёные на каждом шаге; финальный прогон по всему решению —
  743 теста, 743 пройдено, 0 упало.
- Решения/примечания:
  - Именование тестов строго `MethodName_Condition_ExpectedResult`, один
    тестовый метод — один сценарий (без объединения нескольких Assert на
    разные ветки в одном тесте).
  - Для сервисов с внешними FK-зависимостями конструктор фикстуры настраивает
    "счастливый путь" по умолчанию (моки репозиториев возвращают валидные
    связанные сущности), а каждый негативный тест точечно переопределяет один
    мок — это соответствует правилу «не выдумывай» и держит тесты читаемыми.
  - Для `ProductListingService.GetAllAsync(pageNumber, pageSize)` (единственный
    паджинированный метод в проекте) отдельно протестированы разбиение на
    страницы, вторая страница, пустой репозиторий и невалидные pageNumber/
    pageSize.
  - `DeliveryStatus` в тестах использует реальные значения энума
    (`Pending/Assigned/PickedUp/InDelivery/Delivered/Cancelled`) — при первом
    заходе ошибочно указал несуществующий `InTransit`, поймано компилятором
    сразу же и исправлено до коммита.
  - `CustomerType` — аналогично, реальные значения `Retail/Wholesale` (не
    `Individual`), исправлено сразу при первой ошибке компиляции.

## Analytics Dashboard backend (Admin + Farmer) — frontend пропущен

- Что сделано: по `analytics-dashboard-prompt.md` перед реализацией обнаружены
  два несоответствия между промптом и реальным состоянием проекта — заданы
  вопросы пользователю через AskUserQuestion, получены явные ответы:
  1. В проекте вообще нет системы аутентификации (JWT/Identity — по коду в
     `Program.cs` это ещё не реализованный "Этап 2"). Решение пользователя:
     эндпоинты без `[Authorize]`, `farmerId` — query-параметр, с `TODO`-
     комментарием в контроллере на добавление `[Authorize(Roles = "Farmer")]`
     и чтения из claims, когда появится Auth.
  2. `Frontend/Front` — пустая папка, ни React-проекта, ни `package.json`, ни
     единой страницы, ни навигации, куда можно было бы добавить пункт меню.
     Решение пользователя: frontend-часть промпта полностью пропущена,
     сделан только backend.
- Реализован `IAnalyticsRepository`/`AnalyticsRepository` — отдельный
  (не generic-CRUD) репозиторий с агрегациями через LINQ прямо на
  `AppDbContext` (`GroupBy`/`Sum`/`Count` на уровне БД), т.к. `IAnalyticsService`
  в `Application` не может напрямую использовать `AppDbContext`
  (`MarketTJ.Application.csproj` не ссылается на EF Core/Infrastructure —
  Clean Architecture в проекте это явно запрещает). Это единственное
  архитектурное отступление от буквального текста промпта ("агрегации
  делать через LINQ на DbContext" — сделано, но на уровне Infrastructure,
  а не Application, по установленному в проекте паттерну Repository).
- Файлы: `Application/Dto/AnalyticsDto/{AdminDashboardDto,FarmerDashboardDto,
  TopSellingProductDto,OrderStatusCountDto,MonthlyRevenueDto}.cs`,
  `Application/Interfaces/Repositories/IAnalyticsRepository.cs`,
  `Application/Interfaces/Services/IAnalyticsService.cs`,
  `Application/Services/AnalyticsService.cs`,
  `Infrastructure/Persistence/Repositories/AnalyticsRepository.cs`,
  `WebApi/Controllers/AnalyticsController.cs`, DI в
  `Application/DependencyInjection.cs` и `Infrastructure/DependencyInjection.cs`.
- Решения/примечания:
  - "Успешная" выручка = `Order.TotalAmount` только по заказам со
    `Status == Completed` (раздел 10.4 ТЗ), а не по `Payment` — в проекте
    ещё нет кода, который бы автоматически создавал `Payment` при доставке/
    завершении заказа (раздел 8.24 предполагает это на будущее), поэтому
    агрегация по `Payment` всегда возвращала бы 0 независимо от реальных
    продаж. `RevenueThisMonth`/`RevenueByMonth` считаются по `CompletedAt`
    (когда заказ реально завершился), а не по `CreatedAt`.
  - `RevenueByMonth` — 6 последних месяцев (нижняя граница диапазона 6–12 из
    промпта), с явным заполнением нулями месяцев без завершённых заказов —
    чтобы фронтенду в будущем не приходилось самому достраивать ряд для графика.
  - `TopSellingProducts` группируется по `OrderItem.ProductName` (снимок
    названия на момент заказа), а не по `Product`/`ProductListing` — так
    показатель не ломается, если товар/объявление потом переименовали или
    удалили.
  - Проверено вручную реальным запуском `dotnet run` + `curl` против настоящей
    PostgreSQL: `GET /api/analytics/admin/dashboard` вернул реальные цифры
    (4 пользователя, 1 фермер/1 покупатель/1 курьер, 16 активных объявлений,
    0 заказов — соответствует сидеру, который заказы не создаёт);
    `GET /api/analytics/farmer/dashboard?farmerId=1` — 16 своих объявлений,
    `averageRating: null` (отзывов нет); `farmerId=999` → `NotFound`;
    `farmerId=0` → `Validation`.
  - `dotnet build` — 0 ошибок, 0 предупреждений; `dotnet test` — все 743
    ранее написанных теста по-прежнему зелёные (регрессий не внесено).

## Controllers для всех сервисов + Exception/RequestLogging middleware

- Что сделано: у 30 из 32 `I{Entity}Service` не было контроллера (были только
  `AiAssistantController`/`AnalyticsController`). `ICacheService` — не CRUD-
  сервис (généric-кэш, `GetAsync<T>/SetAsync<T>/RemoveAsync`), контроллер ему
  не нужен и не создавался.
- Вынес общий маппинг `Result<T> -> IActionResult` (дублировался в
  `AiAssistantController` и `AnalyticsController`) в новый
  `Controllers/ApiControllerBase.cs` (`[ApiController]`, метод
  `HandleResult<T>`), оба существующих контроллера переведены на него.
- Создано 30 `Controllers/{Entity}Controller.cs` — по одному на каждый CRUD-
  сервис, роуты `api/{entity-plural-kebab-case}`, 5 стандартных эндпоинтов
  (`GET /`, `GET /{id}`, `POST /`, `PUT /{id}`, `DELETE /{id}`). Один сервис
  (`IProductListingService.GetAllAsync(pageNumber, pageSize)`) отличается от
  стандартного CRUD пагинацией — в `ProductListingController` `GET /`
  принимает `[FromQuery] pageNumber = 1, pageSize = 10`. Дополнительных
  методов сверх стандартных пяти ни в одном из 30 интерфейсов не оказалось —
  добавлять отдельно было нечего. Несовпадений DTO/сигнатур не найдено —
  все 30 интерфейсов после прошлых сессий полностью соответствуют паттерну.
- Middleware (`WebApi/Middleware/`):
  - `ExceptionHandlingMiddleware` — ловит необработанные исключения, логирует
    через `ILogger`, отдаёт `{ statusCode: 500, message }`; в Development —
    `ex.Message`, в остальных окружениях — общая фраза (через
    `IWebHostEnvironment.IsDevelopment()`).
  - `RequestLoggingMiddleware` — логирует каждый запрос (`Method`, `Path`,
    итоговый `StatusCode`, время в мс через `Stopwatch`) на уровне
    `Information`.
  - В `Program.cs`: `ExceptionHandlingMiddleware` подключён первым в pipeline
    (до Swagger/HTTPS-redirect/статики/авторизации), `RequestLoggingMiddleware`
    — сразу за ним.
- Проверено вручную (`dotnet run` + `curl`): `GET /api/users` → 200 с реальными
  4 пользователями из seed; `GET /api/users/99999` → 404
  `{"message":"Пользователь не найден"}`; `GET /api/categories` → 200 с 6
  категориями; `POST /api/users` с битым JSON → framework сам вернул 400
  ProblemDetails (это штатное поведение `[ApiController]`, до нашего
  `ExceptionHandlingMiddleware` не дошло — исключения не было). Лог сервера
  подтвердил, что `RequestLoggingMiddleware` отработал на всех запросах,
  включая этот 400: `GET /api/users responded 200 in 103ms`,
  `GET /api/users/99999 responded 404 in 38ms`,
  `POST /api/users responded 400 in 41ms`. Настоящий 500 organically не
  воспроизводился — все 30 сервисов уже оборачивают код в try/catch и
  возвращают `Result.Fail(InternalServerError)` изнутри (по дизайну
  прошлых сессий), поэтому исключение до middleware в норме не долетает;
  специально ломать код/БД, чтобы искусственно вызвать 500, не стал —
  корректность порядка регистрации в pipeline подтверждена сборкой и кодом.
- `dotnet build` — 0 ошибок, 0 предупреждений; `dotnet test` — все 743 теста
  по-прежнему зелёные.
