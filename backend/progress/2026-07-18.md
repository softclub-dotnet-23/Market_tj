## Unit-тесты для всех 30 Application/Services/*Service.cs

- Что сделано: по `generate-service-tests-prompt.md` — до начала задачи
  `Application.Tests/Services/` был полностью пуст (только `.gitkeep`, тестов
  не было вообще ни для одного сервиса, «эталонного» `DiscountRuleServiceTests`
  не существует). Тест-фреймворк в проекте — xUnit (`xunit` 2.9.3), mocking-
  framework отсутствовал — добавлен `Moq` 4.20.72 (пакет уже был в локальном
  NuGet-кэше) в `MarketTJ.Application.Tests.csproj`. Для каждого из 30
  `{Entity}Service.cs` написан `Application.Tests/Services/{Entity}ServiceTests.cs`:
  зависимости (репозитории, `ILogger<T>`) замоканы через `Mock<T>`, сама логика
  сервиса не мокается. Покрыты: `GetAllAsync` (список/пусто/исключение),
  `GetByIdAsync` (найдено/NotFound/исключение), `CreateAsync` (валидные данные,
  отдельный тест на каждое правило валидатора, отдельные тесты на
  FK/уникальность/бизнес-правила, исключение репозитория), `UpdateAsync`
  (аналогично + запись не найдена), `DeleteAsync` (успех/NotFound/исключение).
  Кэш-сценарий из промпта не применялся ни к одному сервису — ни один из 30
  `{Entity}Service` не принимает `ICacheService`/`IMemoryCache` в конструкторе
  (кэширование, упомянутое в разделе 8.17 ТЗ для AppSetting, в реальный код ещё
  не добавлено).
- Файлы: 30 файлов `*ServiceTests.cs` (по одному на сервис) +
  `MarketTJ.Application.Tests.csproj` (добавлен `PackageReference Moq`).
- Порядок реализации (один коммит на сервис — `git commit -m "add tests for
  {Entity}Service"`): User, FarmerProfile, CustomerProfile, CourierProfile,
  Category, Product, ProductListing, ProductImage, CartItem, Order, OrderItem,
  Delivery, Review, Notification, Conversation, ChatMessage, AppSetting,
  FarmerDocument, AuditLog, ReportedListing, RefundRequest, DeliveryZone,
  Commission, Payment, Favorite, FarmerStaffMember, SupportTicket,
  SupportMessage, DeliverySlot, DailySalesSnapshot.
- После каждого файла — `dotnet test MarketTJ.Application.Tests.csproj`,
  все тесты зелёные на каждом шаге; финальный прогон по всему решению —
  743 теста, 743 пройдено, 0 упало.
- Решения/примечания:
  - Именование тестов строго `MethodName_Condition_ExpectedResult`, один
    тестовый метод — один сценарий (без объединения нескольких Assert на
    разные ветки в одном тесте).
  - Для сервисов с внешними FK-зависимостями конструктор фикстуры настраивает
    "счастливый путь" по умолчанию (моки репозиториев возвращают валидные
    связанные сущности), а каждый негативный тест точечно переопределяет один
    мок — это соответствует правилу «не выдумывай» и держит тесты читаемыми.
  - Для `ProductListingService.GetAllAsync(pageNumber, pageSize)` (единственный
    паджинированный метод в проекте) отдельно протестированы разбиение на
    страницы, вторая страница, пустой репозиторий и невалидные pageNumber/
    pageSize.
  - `DeliveryStatus` в тестах использует реальные значения энума
    (`Pending/Assigned/PickedUp/InDelivery/Delivered/Cancelled`) — при первом
    заходе ошибочно указал несуществующий `InTransit`, поймано компилятором
    сразу же и исправлено до коммита.
  - `CustomerType` — аналогично, реальные значения `Retail/Wholesale` (не
    `Individual`), исправлено сразу при первой ошибке компиляции.
