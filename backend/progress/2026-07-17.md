## 12:00 — Сгенерированы Create/Update/Delete DTO для всех 30 сущностей

- Что сделано: по запросу пользователя (пример стиля — `IMenuCategoryRepository` из чужого проекта) созданы 90 файлов DTO в `backend/MarketTJ.Application/Dto/{Entity}Dto/` — по 3 файла (`Create/Update/Delete{Entity}Dto.cs`) на каждую из 30 сущностей. Create = все скалярные поля/FK кроме Id/CreatedAt/UpdatedAt/IsDeleted/DeletedAt/navigation; Update = Create + Id; Delete = только Id. `Application/Dto/` был полностью пустым до этого шага (папок-подпапок не существовало вообще, хотя задача предполагала, что они уже есть) — создавал подпапки сам, по числу существующих repositories/entities (30).
- Файлы: 90 файлов DTO (`UserDto/`, `FarmerProfileDto/`, `CustomerProfileDto/`, `CourierProfileDto/`, `CategoryDto/`, `ProductDto/`, `ProductListingDto/`, `ProductImageDto/`, `CartItemDto/`, `OrderDto/`, `OrderItemDto/`, `DeliveryDto/`, `ReviewDto/`, `NotificationDto/`, `ConversationDto/`, `ChatMessageDto/`, `AppSettingDto/`, `FarmerDocumentDto/`, `AuditLogDto/`, `ReportedListingDto/`, `RefundRequestDto/`, `DeliveryZoneDto/`, `CommissionDto/`, `PaymentDto/`, `FavoriteDto/`, `FarmerStaffMemberDto/`, `SupportTicketDto/`, `SupportMessageDto/`, `DeliverySlotDto/`, `DailySalesSnapshotDto/`).
- Решения/примечания:
  - Папок с "не найденной сущностью" (правило 6 задачи) — не было, т.к. все 30 подпапок создавались из уже существующего списка Entities/Repositories.
  - `CreateUserDto`/`UpdateUserDto` содержат поле `PasswordHash` (не `Password`) — буквально как в entity, хотя в реальном use-case обычно передаётся сырой пароль для хэширования сервисом; не стал переименовывать/менять семантику самостоятельно (правило "не выдумывай новые поля").
  - `Order`/`AppSetting`/`FarmerDocument` и т.п. — в Create DTO включены поля типа `AcceptedAt`/`CompletedAt`/`CancelledAt`/`UploadedAt`/`ReviewedAt`, которые по смыслу вряд ли задаются при создании — но это не `CreatedAt`/`UpdatedAt` буквально, а правило исключало только `Id, CreatedAt, UpdatedAt, IsDeleted, DeletedAt` — оставил как есть по букве задания.
  - Каждая папка закоммичена отдельно (`git add` + `git commit -m "add dtos for {Entity}"`), 30 коммитов подряд.
  - `dotnet build MarketTJ.slnx` после всех 30 коммитов — 0 ошибок, 0 предупреждений.

## 18:00 — Database Seeder (Infrastructure/Persistence/Seed/Seeder.cs)

- Что сделано: по `seeder-prompt.md` (уточнённая версия после первого CafeCRM-примера,
  который был отклонён/прерван пользователем) перед написанием кода заново
  прочитаны все 30 Entities/Enums и текущий `AppDbContext` — за это время внешне
  снова поменялись `Order`/`Review`/`CartItem` (опять на CustomerProfile/FarmerProfile
  вместо User), появились готовые реализации всех 30 репозиториев и `RedisCacheService`
  (не мои — уже были в проекте на момент чтения). Seeder заполняет: Users (по одному
  на роль UserRole: Admin/Farmer/Customer/Courier, раздел 22), FarmerProfile
  (Verified — иначе по разделу 10.1 нельзя создать Active-объявление),
  CustomerProfile, CourierProfile, Categories (точный список раздела 22),
  Products (раздел 8.6 + типичные для таджикского рынка в тех же 6 категориях),
  ProductListings (16 штук от тестового фермера). `DiscountRule` в проекте нет —
  SeedDiscountRulesAsync не создавал (условие самого промпта).
- Файлы: `MarketTJ.Infrastructure/Persistence/Seed/Seeder.cs` (новый),
  `MarketTJ.WebApi/Program.cs` (добавлен вызов `Seeder.SeedAsync` после Migrate),
  `MarketTJ.Infrastructure.csproj` (добавлен пакет `BCrypt.Net-Next` — устоявшегося
  способа хеширования паролей в проекте не было, Auth ещё не реализован).
- Решения/примечания:
  - Пароль Admin — из переменной окружения `ADMIN_PASSWORD` с fallback на
    дефолт для локальной разработки (раздел 22 ТЗ: "Password: задаётся через
    environment variable"), пароли остальных тестовых пользователей — захардкожены
    (это seed/demo-аккаунты, не прод).
  - Дедуп-проверки — через `context.Xxx.AnyAsync()`/`FirstOrDefaultAsync()`
    напрямую (у `IUserRepository` и остальных нет специфичных методов вроде
    `GetByEmailAsync` — только generic CRUD), как и требует сам промпт
    ("AnyAsync-проверка перед вставкой").
  - `IgnoreQueryFilters()` при проверке email — у `User` есть soft-delete
    query filter (`!IsDeleted`), а unique-индекс на Email не отфильтрован,
    поэтому проверка без `IgnoreQueryFilters()` могла бы пропустить
    существующую (пусть и мягко удалённую) запись и упасть на unique constraint.
  - **Проверено реальным запуском**: `dotnet ef database update` — база уже
    актуальна (миграция была применена ранее). Затем `dotnet run` дважды —
    первый раз (по таймауту) успел досеять данные без исключений, второй
    прогон подтвердил идемпотентность (все `AnyAsync`/`EXISTS`-проверки вернули
    true, ни одного `INSERT`). Прямую проверку через `psql` выполнить не удалось —
    `psql` не установлен в этом окружении; опирался на лог реальных SQL-запросов
    EF Core к настоящей PostgreSQL (без ошибок).
  - `git commit -m "add database seeder"` — как просил промпт (без разбивки по сущностям).
