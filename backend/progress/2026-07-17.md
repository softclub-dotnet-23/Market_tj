  SeedDiscountRulesAsync не создавал (условие самого промпта).
- Файлы: `MarketTJ.Infrastructure/Persistence/Seed/Seeder.cs` (новый),
  `MarketTJ.WebApi/Program.cs` (добавлен вызов `Seeder.SeedAsync` после Migrate),
  `MarketTJ.Infrastructure.csproj` (добавлен пакет `BCrypt.Net-Next` — устоявшегося
  способа хеширования паролей в проекте не было, Auth ещё не реализован).
- Решения/примечания:
  - Пароль Admin — из переменной окружения `ADMIN_PASSWORD` с fallback на
    дефолт для локальной разработки (раздел 22 ТЗ: "Password: задаётся через
    environment variable"), пароли остальных тестовых пользователей — захардкожены
    (это seed/demo-аккаунты, не прод).
  - Дедуп-проверки — через `context.Xxx.AnyAsync()`/`FirstOrDefaultAsync()`
    напрямую (у `IUserRepository` и остальных нет специфичных методов вроде
    `GetByEmailAsync` — только generic CRUD), как и требует сам промпт
    ("AnyAsync-проверка перед вставкой").
  - `IgnoreQueryFilters()` при проверке email — у `User` есть soft-delete
    query filter (`!IsDeleted`), а unique-индекс на Email не отфильтрован,
    поэтому проверка без `IgnoreQueryFilters()` могла бы пропустить
    существующую (пусть и мягко удалённую) запись и упасть на unique constraint.
  - **Проверено реальным запуском**: `dotnet ef database update` — база уже
    актуальна (миграция была применена ранее). Затем `dotnet run` дважды —
    первый раз (по таймауту) успел досеять данные без исключений, второй
    прогон подтвердил идемпотентность (все `AnyAsync`/`EXISTS`-проверки вернули
    true, ни одного `INSERT`). Прямую проверку через `psql` выполнить не удалось —
    `psql` не установлен в этом окружении; опирался на лог реальных SQL-запросов
    EF Core к настоящей PostgreSQL (без ошибок).
  - `git commit -m "add database seeder"` — как просил промпт (без разбивки по сущностям).
